# 4領域抽出 + 線予測 マルチタスク学習 設計方針案

## 1. 目的

本設計の目的は、頸椎CTの2Dスライスから以下を同時に予測するマルチタスク学習系を構築することである。

- **領域抽出（主タスク）**
  - 4領域 + 背景のセマンティックセグメンテーション
- **線予測（副タスク）**
  - 4本の境界線に対応するヒートマップ回帰

最終的には、4領域マスクを用いた領域別骨折検出およびVAIリスク評価へ接続することを前提とする。

---

## 2. 現状整理

### 2.1 既存実装で確認できたこと

現在のリポジトリには、線予測専用の 2D U-Net 系実装が存在する。

- 入力: CT画像 + 椎体マスク（2ch）
- 出力: 4ch heatmap
- 損失: MSE を主軸
- 推論: ヒートマップからモーメント法で直線復元

また、dataset 側ではすでに以下が扱える状態になっている。

- `gt_masks/slice_xxx.png` の読み込み
- `gt_region_mask`
- `has_gt_region_mask`

つまり、**線タスク専用実装とは別にパイプラインを作り直さなくても、領域教師を用いた multitask 学習へ拡張可能な状態にある**。

### 2.2 ただし、そのまま line-only を拡張するだけでは不十分

今回の主眼は「線を少し良くすること」ではなく、**4領域抽出を成立させ、その上で線予測も安定化すること**にある。

そのため、設計の優先順位は以下とする。

1. **領域抽出が主役**
2. 線予測は領域理解を補助・補完する副タスク
3. 既存資産は参考にするが、モデル容量や構造は必要なら見直す

---

## 3. 設計の基本方針

### 方針1. 主タスクは領域抽出

マルチタスクの中心は、以下の 5-class segmentation とする。

- 0: 背景
- 1: 左横突孔領域
- 2: 椎体中心部
- 3: 右横突孔領域
- 4: 後方要素

線予測は、上記領域の境界を表現する補助タスクとして扱う。

### 方針2. 2D のまま進める

現段階では以下の理由から 2D ベースで統一する。

- 既存データセット・教師が 2D スライス前提
- 既存 line-only 系との比較が容易
- 症例数がまだ十分多くない
- 軸補正済みスライスを直接扱う現行ワークフローと整合的

### 方針3. Tiny は比較用、主力はもう一段強いモデル

`TinyUNet` は line-only ベースラインとしては有用だが、4領域 segmentation を主役にする本命機としては容量不足の懸念がある。

特に横突孔周辺は

- 面積が小さい
- 左右で混同しやすい
- 骨折で形が崩れやすい

ため、**segmentation を成立させるには tiny より一段強い表現力が必要**と考える。

### 方針4. 椎体 one-hot 条件付けは本命では採用しない

椎体種別（C1〜C7）の one-hot conditioning は、既存検討で性能悪化の可能性が示唆されているため、本命系では一旦採用しない。

理由:

- 入力の椎体マスク自体に形態情報が含まれる
- 小規模データで shortcut 学習を誘発しやすい
- Multitask 化でまず必要なのは解剖理解の共有であり、クラス埋め込みの追加ではない

## 4. 推奨アーキテクチャ案

## 4.1 本命案

**2D ResUNet-medium + shared encoder + dual decoder** を推奨する。

### 入力

- CT画像 1ch
- 椎体マスク 1ch
- 合計 2ch

### 出力

#### Segmentation head

- 5ch logits
  - 背景
  - 左横突孔
  - 椎体中心部
  - 右横突孔
  - 後方要素

#### Line head

- 4ch heatmap
  - line_1
  - line_2
  - line_3
  - line_4

### Backbone

- encoder depth: 4
- channel幅の例: `32, 64, 128, 256`
- block: residual block
- normalization: GroupNorm を第一候補
- dropout: 0.05〜0.10 程度

### Decoder構造

#### 推奨

- **shared encoder**
- **seg decoder**
- **line decoder**

すなわち、encoder は共有し、decoder はタスクごとに分ける。

### dual decoder を採用する理由

segmentation と line heatmap は、必要な出力特性が異なる。

- segmentation: 面の塗り分け・連続領域の再構成
- line heatmap: 細い境界の位置と方向の回帰

この2つを最後の 1x1 conv だけで分けると干渉しやすい。逆に encoder まで完全分離するとデータ効率が悪い。

そのため、

- **encoder は共有して解剖表現を学ぶ**
- **decoder は分けて出力特性を分離する**

という構成が最も妥当と考える。

---

## 4.2 発展案

segmentation decoder の高解像度特徴を line decoder に一部入力し、

- 面理解 → 境界予測

の流れを強める設計も候補である。

ただし最初の比較実験では複雑化しすぎるため、第一段階では

- shared encoder
- 独立 dual decoder

から始める。

---

## 4.3 比較用の軽量案

比較実験として以下を置く価値はある。

### 軽量 multitask baseline

- TinyUNet ベース
- shared trunk + 2 head
- 出力のみ 5ch + 4ch に分岐

これは「small model でも multitask 化だけで改善するか」を見る比較対象として使う。

ただし **本命ではない**。

## 5. 損失設計

現段階では、**難しい損失は入れない**方針とする。  
特に、線予測については

- angle loss
- rho loss
- moment consistency loss
- 境界と領域の整合性 loss

のような追加幾何損失は使わない。

目的は、まず

1. multitask 化そのものが有効か
2. 主タスクである領域抽出が成立するか
3. 線予測が悪化せず維持または改善するか

を、できるだけ単純な条件で検証することである。

### 5.1 線予測 loss

線予測は、既存 line-only 系と同様に **heatmap MSE のみ**を用いる。

\[
L_{line} = \mathrm{MSE}(\hat{H}, H)
\]

- 出力: 4ch heatmap
- GT: 4本の境界線から生成したガウシアンヒートマップ
- 活性化: sigmoid 後に MSE

**追加の線損失は使わない。**

### 5.2 領域抽出 loss

領域抽出は、まずは **CrossEntropyLoss のみ**で開始する。

\[
L_{seg} = L_{CE}
\]

理由:

- まずは最小構成で成立性を確認したい
- Dice まで入れると、multitask の効果と loss 設計の効果が混ざる
- 4領域 segmentation が成立するかを最も単純に検証したい

必要であれば後続段階で Dice を追加するが、**最初の本命実験では入れない**。

### 5.3 総損失

\[
L = L_{line} + \alpha L_{seg}
\]

ここで `alpha` は segmentation loss が強すぎないように調整する。

初期案:

- `alpha = 0.1` から開始
- 必要に応じて `0.3` まで調整

### 5.4 部分教師への対応

`has_gt_region_mask` が `True` のサンプルのみ segmentation loss を計算する。

- 領域教師あり: `L_line + alpha * L_seg`
- 領域教師なし: `L_line`

これにより、領域マスク未整備のサンプルも線学習には活用できる。

## 6. 学習戦略

### 6.1 初期段階の方針

初期段階では、**難しい損失を追加せず、単純な multitask 学習**に限定する。

- line: MSE only
- region: CrossEntropy only
- total: 加重和のみ

### 6.2 学習開始方法

まずは以下の2条件を比較する。

#### 条件A: 線予測単独
- 既存 line-only baseline

#### 条件B: multitask
- line MSE
- region CE
- shared encoder + dual decoder

この比較で、

- segmentation を足すことで line が良くなるか
- 少なくとも line が壊れないか
- segmentation が成立するか

を確認する。

### 6.3 後回しにするもの

以下は**最初の段階では使わない**。

- angle loss
- rho loss
- line parameter loss
- moment-based loss
- seg-line consistency loss
- Dice loss
- boundary-aware loss
- focal loss

これらは、単純 multitask での挙動を確認してから追加検討する。
