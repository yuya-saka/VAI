# 線予測角度ずれ問題 - 最終調査レポート (2026-03-19)

## 問題の概要

**症状:** モデルが予測する線の角度がGT線と8〜28度ずれる

**影響範囲:** テストデータ全体で平均14.5度の角度誤差

---

## 調査の経緯

### Phase 1: 座標系の検証 ❌ (否定)
**仮説:** 座標変換（画像座標系 ⟷ 数学座標系）にバグがある

**検証結果:**
- `line_losses.py` と `line_detection.py` の座標グリッドは完全に一致
- GTと同じ形状のヒートマップからは**誤差0.2度以内**で角度を抽出できた
- 座標系は正しく統一されている

**結論:** ❌ 座標系の問題ではない

---

### Phase 2: ヒートマップの滑らかさ検証 ❌ (誤った分析)
**仮説:** ヒートマップが滑らかな直線を形成していない

**検証方法:**
- 行ごとの最大値インデックスの標準偏差を計算
- 初期結果: std = 85〜117（極めて高い）

**問題点:**
- **全224行を対象に計算していた**（ヒートマップが存在しない行も含む）
- ヒートマップの有効範囲のみで再計算すると **std = 1.5〜4.2**
- これは十分滑らかな値

**結論:** ❌ 初期分析は誤り。ヒートマップは有効範囲内では十分滑らか

---

### Phase 3: 真の原因の特定 ✅

## 発見された真の問題

### 🔴 ヒートマップの有効範囲が極端に短い

**サンプル: sample22_C1_slice038**

| Line | 有効行数 | 全体に占める割合 | 閾値以上ピクセル比率 | 角度誤差 |
|------|----------|------------------|---------------------|----------|
| line_1 | 40 / 224 | **18%** | 1.29% | **41.05度** ❌ |
| line_2 | 19 / 224 | **8.5%** | 1.37% | 2.80度 ✅ |
| line_3 | 39 / 224 | **17%** | 1.20% | **28.83度** ❌ |
| line_4 | 15 / 224 | **6.7%** | 0.81% | 2.25度 ✅ |

**基準:**
- 有効行数（max > 0.15）が全体の **6.7%〜18%** のみ
- 閾値以上のピクセルは全画像の **0.8%〜1.4%** のみ
- 質量（M00）が 487〜705 と少ない

### 🔴 モーメント法が不安定になる原因

**モーメント法の特性:**
```
主軸方向 = 2次モーメントの固有ベクトル
mu20 = Σ(heatmap * (x - cx)^2) / M00
mu02 = Σ(heatmap * (y - cy)^2) / M00
mu11 = Σ(heatmap * (x - cx) * (y - cy)) / M00
```

**問題:**
1. **有効範囲が短い** → サンプル点が少ない → 主軸方向の推定が不安定
2. **質量が少ない** → ノイズの影響を受けやすい
3. **細い分布** → 長軸と短軸の比が小さい → 方向の曖昧さが増す

### ✅ スケルトン（二値化後）では正しい

**重要な観察:**
- **二値化（threshold=0.15）後のスケルトンとGT線はほぼ一致**
- つまり、ヒートマップの「形状」は大体正しい
- しかし「質量分布の広がり」が不足している

## 根本原因

### モデルが「細く短いヒートマップ」を学習している

**期待される学習:**
- 広い範囲に滑らかなガウシアン分布（sigma=2.5）
- 直線全体をカバー

**実際の学習結果:**
- 極めて局所的な分布（全体の6-18%の行のみ）
- ピーク周辺のみが高い値
- 直線全体には広がっていない

### なぜこうなったか？

**仮説1: MSE損失の限界**
- MSE損失は「ピークの位置」を合わせることに集中
- 分布の「広がり」を学習するインセンティブがない
- 結果: ピークは合うが、分布は狭い

**仮説2: Sigmoidの影響**
- `sigmoid(model(x))` で出力を[0,1]に制限
- 小さな値が抑制されやすい
- GT（sigma=2.5のガウシアン）より狭い分布になりやすい

**仮説3: データ拡張の影響**
- 回転・スケール変換時の補間がガウシアンの広がりを損なう
- 変換後に再計算していない

---

## 解決策

### 🎯 優先度 高: ヒートマップの広がりを改善

#### 1. 損失関数の変更
```python
# 現在: MSE のみ
loss = F.mse_loss(pred, gt_heatmap)

# 提案: 分布を保つ損失を追加
kl_loss = F.kl_div(pred.log(), gt_heatmap, reduction='batchmean')
total_loss = mse_loss + lambda_kl * kl_loss
```

**効果:**
- ガウシアン分布の形状を保つ
- 広がりを学習する

#### 2. GTヒートマップのsigma値を大きくする
```yaml
# 現在
sigma: 2.5

# 提案
sigma: 4.0  # より広い分布
```

**効果:**
- より広い範囲に分布
- モーメント計算が安定する

#### 3. マルチスケール監視
```python
# 異なるスケールでヒートマップを監視
for scale in [1.0, 0.5, 0.25]:
    pred_scaled = F.interpolate(pred, scale_factor=scale)
    gt_scaled = F.interpolate(gt, scale_factor=scale)
    loss += F.mse_loss(pred_scaled, gt_scaled)
```

**効果:**
- 粗いスケールで大局的な形状を学習
- 細かいスケールでピークの位置を学習

### 🎯 優先度 中: 出力層の改善

#### 4. Softmax出力（チャンネルごと）
```python
# 現在
out = torch.sigmoid(model(x))

# 提案
out = torch.softmax(model(x).view(B, C, -1), dim=2).view(B, C, H, W)
```

**効果:**
- 各チャンネル内で質量が保存される
- より明確なピーク形成

#### 5. 正則化項の追加
```python
# ヒートマップの空間的連続性を促す
smoothness_loss = F.mse_loss(pred[:,:,1:,:], pred[:,:,:-1,:]) + \
                  F.mse_loss(pred[:,:,:,1:], pred[:,:,:,:-1])
```

**効果:**
- 滑らかな分布を促進

### 🎯 優先度 低: データ拡張の見直し

#### 6. 拡張後のヒートマップ再計算
```python
# 拡張後、変換された座標でヒートマップを再生成
# 補間だけでなく、正しいガウシアンを再計算
```

## 結論

**問題の本質:**
- ❌ 座標系の問題ではない
- ❌ ヒートマップの滑らかさの問題でもない
- ✅ **ヒートマップの有効範囲が極端に短く、質量分布が少ないことが問題**

**角度ずれが発生するメカニズム:**
1. モデルが細く短いヒートマップを出力
2. モーメント法での2次モーメント計算のサンプル点が不足
3. 主軸方向の推定が不安定になる
4. 結果として角度が8〜28度ずれる

**なぜ2本は正確で2本はずれるのか？**
- line_2, line_4: たまたま短い範囲でも方向が安定
- line_1, line_3: 短い範囲では方向の推定が不安定

**解決の方向性:**
- ヒートマップをより広い範囲に分布させる
- KL divergence などで分布の形状を保つ
- sigma値の増加を検討
