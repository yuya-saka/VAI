### 5. アーキテクチャ変更方針の議論（実装未着手）

**課題**: 現行 `shared encoder + dual decoder` では seg decoder が line 情報を直接参照しない。「線を使って seg 境界を改善する」構造になっていない。

**採用方針**: `shared encoder + late-branch shared decoder`

```
Encoder → DecoderTrunk (dec4/dec3, 共有)
        → seg_branch (up2/dec2/seg_head, 独立)
        → line_branch (up2/dec2/line_head, 独立)
```

**なぜこの構造か（Codex分析より）**:
- `DecoderTrunk` が seg loss + line loss 両方から勾配を受け取る
- → trunk 内に「輪郭・境界がどこか」という表現が形成される
- → seg_branch がその境界情報を持つ特徴を入力として受け取る
- → seg の境界が sharpen されやすくなる

**なぜ pure dual-head（最終層だけ分岐）では不十分か**:
- line（細い境界）と seg（領域の一貫性）が最高解像度段で競合しやすい
- dec3 以降で分岐することで競合を回避しつつ、共有部で boundary-aware 表現を学習

**Phase 2 オプション（実験後に必要なら）**:
- `boundary = sigmoid(line_logits).detach()` を seg_branch の最終特徴に concat → 3x3 Conv → seg_head
- detach() で seg loss が line branch を逆流しないよう保護

**コード変更スコープ**:
- `Decoder` → `DecoderTrunk` + `DecoderBranch` に分割
- `ResUNet`: `decoder_trunk`, `seg_branch`, `line_branch` の3モジュール構成
- `ResBlock`, `Encoder`, vertebra conditioning は**変更不要**
- 出力インターフェース `{'seg_logits': ..., 'line_heatmaps': ...}` は維持

**Codex分析ファイル**: `.claude/docs/codex/20260412-shared-decoder-design.md`
