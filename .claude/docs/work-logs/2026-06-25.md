## Stage1 GradCAM 可視化 notebook 整備

### 実施内容

`train_models/stage1` の Stage1 classifier 用 GradCAM 可視化を notebook で使える形に整理した。  
最終的に notebook 関連ファイルは `train_models/stage1/notebook/` 配下へ集約した。

### 実装した変更

#### 1. GradCAM 可視化 notebook の作成・移動

- `train_models/stage1/notebook/gradcam_viz.ipynb` を作成
- `train_models/stage1/notebook/gradcam_viz.py` を notebook から import する実装本体として配置
- 以前の `train_models/stage1/gradcam_viz.py` は削除し、notebook 配下へ集約
- notebook を開く場所に依存しないよう、`PROJECT_ROOT` を `train_models/` marker 探索で解決

#### 2. 表示モードの統一

Parameters セルで表示対象を切り替えられるようにした。

```python
# one of: "all", "orientation", "bbox"
DISPLAY_MODE = "bbox"
```

- `all`: 15 classifier planes をすべて表示
- `orientation`: 向き補正に使用した `max_area_forced` plane のみ表示
- `bbox`: bbox がある `bbox_forced` plane のみ表示

`ORIENTATION_ONLY` / `BBOX_ONLY` は互換用フラグとして残したが、基本操作は `DISPLAY_MODE` に統一。

#### 3. bbox 表示対応

- `train_bounding_boxes.csv` の bbox を読み込み
- DICOM pixel 座標から LPS patient 座標へ変換
- classifier plane の 224x224 座標へ射影
- GradCAM 画像上に bbox を赤枠で重ねて表示
- vertebra mask は従来通り緑輪郭で表示

#### 4. 対象スライスなし問題の修正

`DISPLAY_MODE="bbox"` または `"orientation"` のとき、先に `N` 件を取ってから表示対象を探すと `No selected slices` が多発した。

対応として、抽出順を以下に変更した。

1. `CATEGORY` と `FOLD` で候補を抽出
2. `DISPLAY_MODE` に応じて対象スライスを持つサンプルだけに絞る
3. その後で上位 `N` 件を取得

確認例:

- `TP fold0`: 全 157 件
- bbox あり候補: 30 件

#### 5. slider viewer 対応

- `ipywidgets` を dependency に追加
- `uv.lock` を更新
- notebook 下部に slider viewer を追加
- `mode` dropdown の初期値は Parameters セルの `DISPLAY_MODE` を使用
- `Prev` / `Next` ボタンと `Save current PNG` ボタンを追加

### 検証


- notebook 全 code cell の AST parse

2. Parameters セルで `DISPLAY_MODE` を選ぶ

- notebook を開きっぱなしの場合、ブラウザ側に古い notebook 内容が残ることがあるため reload が必要
- `DISPLAY_MODE="bbox"` の場合、bbox が存在する候補だけに絞られるため `N` より少ない件数になる場合がある

## RSNA line-only 可視化の線復元仕様確定

### 背景

`Unet/line_only/predict_rsna.py` と `Unet/line_only/visualize_rsna.py` で、RSNA前処理済みデータに対する4本線ヒートマップ予測は妥当に見える一方、可視化される復元線がヒートマップのピークからずれる問題を調査した。

原因として、以前の junction-based polyline 再構成や端点の個別クリップにより、ヒートマップ moment から得た `centroid + angle` の無限線と、実際に描画・領域分割へ渡す線分が一致しないケースがあった。

### 決定した仕様

RSNAデータに対する線復元は以下で固定する。

1. 各予測ヒートマップから moment/PCA により無限線パラメータを推定する
   - `centroid`
   - `angle_deg`
   - `phi_rad`
   - `rho_normalized`
2. 学習データ `data/dataset/sample*/C*/lines.json` から `line_1`〜`line_4` それぞれの平均線長を計算する
3. RSNA上では、各無限線を line別平均長で線分化する
4. `Pred Lines` の描画と `4 Regions` の `generate_region_mask()` 入力は、同じ平均長線分を使う
5. junction-based polyline 再構成は使わない

計算された平均線長:

```text
line_1 = 38.7 px
line_2 = 39.8 px
line_3 = 39.2 px
line_4 = 40.4 px
```

### 実装した変更

- `Unet/line_only/predict_rsna.py`
  - `compute_average_training_line_lengths()` を追加
  - line別平均長を `detect_line_moments(..., length_px=..., extend_ratio=1.0, clip=False)` に渡すよう変更
  - 出力JSONに `line_length_px` を追加

- `Unet/line_only/visualize_rsna.py`
  - `compute_average_training_line_lengths()` を追加
  - `compute_junction_polylines()` を廃止し、`lines_to_polylines()` で平均長線分を使うよう変更
  - `Pred Lines` と `4 Regions` の入力線分を統一
  - 相対/絶対 `--output-dir` のログ表示で落ちる問題を修正

- `.claude/docs/DESIGN.md`
  - RSNA line restoration policy として仕様を記録済み

### 検証結果

代表例 `1.2.826.0.1.3680043.10001 / C3 / plane=6` で、各ヒートマップピークから復元線までの距離がほぼ0pxになることを確認した。

```text
line_1 dist ≈ 0.05 px
line_2 dist ≈ -0.38 px
line_3 dist ≈ 0.09 px
line_4 dist ≈ 0.00 px
```

また、以下へデフォルト範囲の可視化を再生成済み。

```text
Unet/outputs/rsna_line_vis
```

生成条件:

```bash
uv run python Unet/line_only/visualize_rsna.py --output-dir Unet/outputs/rsna_line_vis --gpu-id 0
```

デフォルトでは先頭5 study × C1-C7 の35枚を生成。
